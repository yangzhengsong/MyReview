项目一：MCP 与三层记忆架构
1. 你项目中 MCP 用来做什么？
核心观点：MCP 是用来统一 AI Agent 与外部工具/数据源交互的标准化协议。

回答要点：

将知识库检索、数据库查询、API 调用等能力封装为 MCP Server 的 Tools

通过 Resources 机制将文件、Schema 等信息作为上下文暴露给模型

一个 Host 可连接多个 MCP Server，实现多能力源组合调用

话术：

“MCP 主要解决的是工具越多系统越乱的痛点。以前每接一个新工具就要手写工具描述、参数结构、鉴权逻辑，现在统一成 MCP 协议后，新工具只需实现标准接口即可接入。”

2. MCP 的主要作用是什么？
核心观点：标准化、解耦、安全可控。

回答要点：

层面	作用
标准化	统一接口规范，不同来源的工具/数据可同方式接入
解耦	LLM 核心与外部能力分离，独立迭代
安全可控	Host 控制权限，Server 只接收必要上下文
话术：

“MCP 解决的不是‘让模型更聪明’的问题，而是‘让模型周围那圈能力更容易组织起来’的问题。”

3. MCP 还有哪些功能？
核心观点：MCP 是“接口标准 + 功能协议”的组合体，功能不止 Tools。

回答要点：

功能	说明
Resources（资源）	只读数据结构，作为上下文提供给模型（如文件、Schema）
Prompts（提示词模板）	可复用的模板化消息和工作流（如斜杠命令）
Tools（工具）	可供模型执行的函数（如查数据库、调 API）
Elicitation（信息采集）	Server 主动向用户询问额外信息
Sampling（采样）	Server 请求模型生成内容，实现嵌套 AI 交互
4. MCP 有哪些实现方式？
核心观点：基于 JSON-RPC 2.0，主要有四种方式。

回答要点：

SDK 方式：用官方 SDK（Python/TypeScript）快速搭建

stdio 方式：标准输入输出通信，适合本地进程调用

Socket/HTTP 方式：网络通信，适合远程服务

平台集成方式：通过云平台 MCP MarketPlace 接入现成服务

6. 讲一下你项目里的三层记忆架构。
核心观点：三层记忆是为了解决“记忆膨胀”问题——记忆多了塞不进上下文，Token 消耗爆炸。

回答要点：

层级	内容	加载方式
Layer 1：关键记忆	System Prompt、长期记忆文件、今日笔记	每次请求完整加载
Layer 2：近期历史	最近 N 条对话历史（如 50 条）	作为 messages 传入，超出截断
Layer 3：归档记忆	历史日期记忆文件、历史对话	按需加载，不自动送入
话术：

“核心思想是：不是把所有记忆都塞进 LLM 的上下文，而是根据重要性和时效性采用不同的加载策略。”

7. 上下文记忆、checkpoint、长期记忆分别是什么？
核心观点：三者是不同维度的“记忆”，作用和生命周期不同。

回答要点：

概念	说明
上下文/短期记忆	当前会话对话历史，受窗口大小限制，保留最近 N 轮
Checkpoint	特定时刻 Agent 状态的快照，用于恢复、调试、回溯
长期记忆	跨会话持久化的结构化知识，是提取后的事实，不是原始日志
话术：

“对话历史是日志，Checkpoint 是系统快照，长期记忆是结构化知识库。”

8. 为什么长期记忆采用“文件为本，数据库只做索引”？
核心观点：保证可解释性、可维护性与灵活性。

回答要点：

设计	优势
文件为本	人类可读可编辑、可纳入 Git 版本管理、避免向量数据库黑盒
数据库只做索引	快速检索（SQLite 索引）、轻量依赖、索引可重建不丢数据
话术：

“文件存‘是什么’，数据库存‘在哪里’——文件保证记忆的完整性和可读性，数据库保证检索的高效性。两者各司其职，比纯向量数据库方案更透明、更易维护。”

项目二：分库分表
1. 你项目里的分库分表是怎么设计的？
⚠️ 请根据实际项目回答，以下为参考框架：

回答要点：

水平拆分，按业务维度选分片键（如 user_id）

分片数量示例：16 库 × 64 表 = 1024 物理分片

中间件：ShardingSphere-JDBC

设计考量：数据均匀分布、关联查询优化、预留扩容方案

话术：

“我们采用水平拆分，分片键是 user_id。分了 16 个库 × 64 张表，用 ShardingSphere-JDBC 做中间件。设计时重点考虑了三件事：数据分布均匀避免热点，相关表用相同分片键避免跨分片 JOIN，以及预留了扩容方案。”

2. 你说的自定义分片算法具体是什么？
核心观点：根据业务特点自行实现分片路由逻辑，不是简单取模。

回答要点：

接口类型	适用场景
PreciseShardingAlgorithm	精确分片（= 和 IN 查询）
RangeShardingAlgorithm	范围分片（BETWEEN AND）
HintShardingAlgorithm	由代码手动指定分片
参考话术：

“我们实现了 PreciseShardingAlgorithm 接口，分片键是 user_id。算法逻辑是：分片序号 = (user_id >> 4) % 分片总数，让同一用户数据落在同一分片，同时保证分布均匀。范围查询额外实现了 RangeShardingAlgorithm 做路由优化。”

3. 分布式 ID 是自己设计的吗？
核心观点：建议采用成熟方案，不自己造轮子。

参考话术：

“我们没有自己造轮子，采用的是雪花算法（Snowflake）。它是 64 位长整型，由时间戳、机器 ID 和序列号组成。我们在此基础上做了小幅调整，比如增加了数据中心 ID 位段以适应多机房部署。”

4. 你用的分布式 ID 方案是什么？
⚠️ 根据实际方案回答，以下是常见方案速查：

方案	特点	适用场景
UUID	简单但无序、空间大	对排序无要求
雪花算法	64位、趋势递增、高性能	大部分分布式场景
Leaf（美团）	雪花改进 + 号段模式	高并发、高可用
UidGenerator（百度）	雪花改进，解决时钟回拨	高并发
参考话术：

“我们采用雪花算法。选择原因：64 位长整型空间小、趋势递增利于索引性能、不依赖外部服务去中心化高可用。时钟回拨问题通过定期上报机器时间和回拨等待策略规避。”
